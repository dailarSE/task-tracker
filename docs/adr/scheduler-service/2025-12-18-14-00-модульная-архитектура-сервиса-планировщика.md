# ADR-0043: Модульная Архитектура Сервиса-Планировщика (Jobs vs Consumers)

*   **Статус:** Accepted
*   **Дата:** 2025-12-18
*   **Заменяет:** ADR-0040 (Архитектура Сервиса-Планировщика)
*   **Контекст:**
    *   Предыдущая архитектура (ADR-0040) и текущая реализация рассматривали сервис `task-tracker-scheduler` как монолитный компонент, смешивающий логику запуска по расписанию (Scheduling) и логику обработки данных (Processing).
    *   Это приводит к нарушению принципа единственной ответственности (SRP), усложняет тестирование и ограничивает возможности независимого масштабирования (например, нам может понадобиться 1 инстанс для генерации событий и 10 инстансов для их обработки).
    *   Необходимо явно разделить эти ответственности на уровне кода и архитектуры приложения.

*   **Принятое Решение:**

    1.  **Логическое Разделение Доменов:**
        Разделить кодовую базу сервиса на два изолированных логических домена:

        *   **Домен `jobs` (Triggers / Producers):**
            *   **Ответственность:** Инициация процессов по времени (Cron). Управление уникальностью запуска (Distributed Lock). Генерация первичных событий (Events) и их публикация в брокер сообщений.
            *   **Характеристики:** Легковесные компоненты, работают с метаданными, минимальная бизнес-логика.
            *   **Ключевые технологии:** `@Scheduled`, `ShedLock`, `KafkaTemplate`.

        *   **Домен `consumers` (Workers / Processors):**
            *   **Ответственность:** Реакция на события (Event-Driven). Выполнение основной бизнес-логики (обращение к API, агрегация данных, сложные вычисления). Генерация производных событий или команд.
            *   **Характеристики:** Stateless, горизонтально масштабируемые, идемпотентные (по возможности).
            *   **Ключевые технологии:** `@KafkaListener`, Service Beans.

    2.  **Структура Пакетов:**
        Код должен быть организован в соответствующие пакеты для отражения этого разделения:
        *   `com.example.tasktracker.scheduler.job` — компоненты триггеров и управления состоянием джоб.
        *   `com.example.tasktracker.scheduler.consumer` — компоненты-обработчики событий Kafka.

    3.  **Управление Конфигурацией (Profiles):**
        *   Архитектура должна поддерживать запуск приложения в различных режимах через Spring Profiles или свойства конфигурации:
            *   **Режим "Scheduler":** Активны только бины из домена `jobs`.
            *   **Режим "Worker":** Активны только бины из домена `consumers`.
            *   **Режим "All-in-One" (по умолчанию):** Активны все компоненты.
        *   Это реализуется через аннотации `@ConditionalOnProperty` на конфигурационных классах или компонентах.

*   **Последствия:**
    *   **Положительные:**
        *   Четкое разделение ответственности (Separation of Concerns).
        *   Возможность независимого масштабирования (Scale Triggers vs Scale Workers).
        *   Упрощение тестирования (можно тестировать Job и Consumer изолированно).
        *   Повышение отказоустойчивости (падение Worker'а не убивает Scheduler).
    *   **Отрицательные:**
        *   Некоторое увеличение количества классов и пакетов.
        *   Необходимость управления конфигурацией для разных режимов запуска.

*   **Необходимые действия:**
    *   Рефакторинг существующего кода: перенос `DailyTaskReportJobTrigger` и связанных классов в пакет `job`.
    *   Создание пакета `consumer` для будущих обработчиков.
    *   Настройка `@ConditionalOnProperty` для бинов.

# ADR-0044: Процесс "Ежедневный Отчет" (Event-Driven Flow) — Реализация Этапа 1

*   **Статус:** Draft
*   **Дата:** 2025-12-18
*   **Связанные ADR:**
    *   ADR-0043: Модульная Архитектура Сервиса-Планировщика
*   **Контекст:**
    *   Необходимо реализовать первую часть бизнес-процесса формирования ежедневных отчетов: инициацию процесса по расписанию и выборку пользователей, для которых необходимо сформировать отчет.
    *   Процесс должен быть надежным (at-least-once delivery), устойчивым к сбоям (возобновляемым) и не допускать дублирования запуска (single execution instance).

*   **Принятое Решение (Этап 1: Producer Job):**

    1.  **Общая Концепция:**
        *   Продюсер (`DailyReportProducerJob`) запускается по расписанию и итерирует всех пользователей через Backend API.
        *   Для каждого пользователя в Kafka отправляется событие-факт `UserSelectedForDailyReport`.

    2.  **Техническая Реализация Продюсера:**

        *   **2.1. Механизм Запуска и Блокировки:**
            *   Использовать `ShedLock` для гарантии того, что в любой момент времени работает только один экземпляр джобы.
            *   Использовать частый CRON (например, раз в 15 минут), чтобы обеспечить автоматический перезапуск в случае сбоев.

        *   **2.2. Управление Состоянием (State Management):**
            *   Использовать Redis (HASH) для хранения состояния выполнения джобы для конкретной даты отчета.
            *   Структура состояния (`JobState`): `status` (IN_PROGRESS, PUBLISHED, FAILED), `lastCursor` (для пагинации).

        *   **2.3. Алгоритм Выполнения:**
            *   **Шаг 1: Триггер & Блокировка:** Попытка захвата ShedLock. Если занято — выход.
            *   **Шаг 2: Проверка Состояния:** Чтение `JobState` из Redis для `reportDate`.
                *   Если `PUBLISHED` -> Выход (работа уже выполнена).
                *   Если `IN_PROGRESS` / `EMPTY` -> Продолжение работы (с `lastCursor`, если есть).
                *   Если `FAILED` (логическая ошибка) -> Алерт и остановка.
            *   **Шаг 3: Цикл Выборки (Fetch Loop):**
                *   Вызов Backend API (`GET /user-ids`) с текущим курсором.
                *   *Отказ (Сеть/5xx):* Ретрай (Backoff). При исчерпании — прерывание джобы (Exception). Состояние в Redis **НЕ** обновляется.
                *   *Отказ (Данные/4xx):* Сохранение статуса `FAILED` в Redis, Алерт, Остановка.
            *   **Шаг 4: Публикация (Publish):**
                *   Отправка полученных ID в Kafka (событие `UserSelectedForDailyReport`).
                *   *Отказ:* Если хотя бы одно сообщение из пачки не отправлено — **Аварийная остановка** (Exception). Курсор в Redis **НЕ** обновляется (чтобы обеспечить повторную отправку при следующем запуске).
            *   **Шаг 5: Фиксация (Commit):**
                *   Сохранение нового `lastCursor` в Redis со статусом `IN_PROGRESS`.
                *   Продление блокировки (Lock Extension) для предотвращения тайм-аута.
            *   **Шаг 6: Завершение:**
                *   При получении последней страницы данных — сохранение статуса `PUBLISHED` в Redis.

    3.  **Контракт События (Kafka Message):**
        *   **Топик:** `task_tracker.scheduler.events.user_selected_for_report`
        *   **Ключ:** `null` (Round-Robin распределение для балансировки нагрузки на консьюмеров).
        *   **Payload (JSON):**
            ```json
            {
              "userId": 12345,
              "reportDate": "2025-06-20",
              "jobRunId": "uuid-..." 
            }
            ```

*   **Последствия:**
    *   Гарантируется доставка событий "как минимум один раз" (at-least-once). Возможны дубликаты событий при сбое на шаге 5 (Commit), консьюмеры должны быть идемпотентны.
    *   Система автоматически восстанавливается после временных сбоев (сети, брокера) при следующем срабатывании CRON.
    *   Прогресс сохраняется, что позволяет обрабатывать большие объемы данных частями.

# ADR-0045: Процесс "Ежедневный Отчет" — Этап 2 (Consumer & Processing)

*   **Статус:** Accepted
*   **Дата:** 2025-01-07
*   **Связанные ADR:**
    *   ADR-0044: Процесс "Ежедневный Отчет" — Этап 1 (Producer)
    *   ADR-0041: API для Взаимодействия с Сервисом-Планировщиком
    *   ADR-0036: Реализация Kafka Продюсера (контракт EmailTriggerCommand)
    *   ADR-0028: Обсервабилити (Метрики и Трейсинг)

## 1. Контекст
Сервис-планировщик генерирует поток событий `UserSelectedForDailyReportEvent`. Необходимо реализовать компонент-потребитель, который обогатит эти события данными из Backend API и отправит команды на email-рассылку.
Ключевые требования: высокая пропускная способность (batch processing), устойчивость к сбоям API и изолированность ошибок (один "битый" пользователь не должен останавливать рассылку для остальных).

## 2. Принятое Решение

### 2.1. Пакетная Обработка (Batch Processing)
Использовать `BatchListener` в Spring Kafka.
*   Консьюмер вычитывает сообщения пачками (настраивается через `max.poll.records`, например, 500).
*   Для каждой пачки выполняется **один** групповой запрос к Backend API (`fetchTaskReports`).

### 2.2. Стратегия Обработки Ошибок (Error Handling)
Применяется гибридный подход:
1.  **Глобальные сбои (Global Failures):**
    *   *Причина:* Недоступность Backend API, таймаут, разрыв соединения с Kafka.
    *   *Реакция:* Выброс исключения из лисенера -> `DefaultErrorHandler` -> Retry с Exponential Backoff -> (при исчерпании) Весь батч в DLT.
2.  **Частичные сбои (Partial Failures):**
    *   *Причина:* Ошибка маппинга данных конкретного пользователя, логическая ошибка валидации.
    *   *Реакция:* Ошибка перехватывается внутри цикла обработки (`try-catch`). Сбойная запись логируется, метрика ошибок инкрементируется. Обработка остальных записей в батче продолжается. (Опционально: ручная отправка в DLT).

### 2.3. Архитектура Компонентов
Реализация разделяется на 4 слоя ответственности:
1.  **Consumer (Listener):** Слушает Kafka, группирует сообщения по `JobRunId`.
2.  **Processor (Orchestrator):** Управляет бизнес-логикой (Bulk Fetch, MDC context, цикл обработки).
3.  **Mapper:** Чистая функция преобразования `UserTaskReport` -> `EmailTriggerCommand`.
4.  **Publisher:** Инкапсулирует отправку результата в Kafka (Sink topic) и сбор метрик.

### 2.4. Observability
*   **Trace Context:** `correlationId` в команде заполняется текущим `TraceID` из OpenTelemetry (или генерируется новый, если контекст отсутствует).
*   **Метрики:** Обязателен сбор RED-метрик (Rate, Errors, Duration) и метрик насыщения (Batch Size Histogram).

## 3. Последствия
*   Минимизация сетевого трафика (N событий -> 1 запрос к API).
*   Высокая надежность: временные сбои пережидаются, "ядовитые" сообщения изолируются.
*   Четкое разделение ответственности упрощает Unit-тестирование.

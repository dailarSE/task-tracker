# Task Tracker Scheduler Service

Этот сервис является компонентом системы "Task Tracker" и отвечает за выполнение периодических, отказоустойчивых и масштабируемых фоновых задач.

## Основные Обязанности

Сервис разделен на два логических домена (согласно **[ADR-0043](../docs/adr/scheduler-service/2025-12-18-14-00-модульная-архитектура-сервиса-планировщика.md)**):

1.  **Домен `jobs` (Продюсеры):**
    *   **Инициирование процессов:** Запускается по расписанию (например, каждые 15 минут) для старта процесса "Ежедневный Отчет".
    *   **Обеспечение уникальности:** Использует `ShedLock` и `Redis` для гарантии того, что только один экземпляр сервиса в кластере может инициировать процесс.
    *   **Публикация событий:** Итерирует пользователей через `task-tracker-backend` API и публикует события `UserSelectedForDailyReportEvent` в Kafka для дальнейшей обработки. Процесс является возобновляемым.

2.  **Домен `consumers` (Обработчики):**
    *   **Обработка событий:** В пакетном режиме слушает события `UserSelectedForDailyReportEvent` из Kafka.
    *   **Обогащение данных:** Для каждой пачки событий делает один групповой запрос к `task-tracker-backend` API для получения агрегированных отчетов.
    *   **Публикация команд:** Преобразует полученные отчеты в команды `EmailTriggerCommand` и публикует их в соответствующий топик Kafka.

## Поток Данных

Архитектура сервиса построена на принципах Event-Driven и разделения ответственности для обеспечения надежности и масштабируемости.

Полный end-to-end поток данных для процесса "Ежедневный Отчет" можно увидеть на следующей диаграмме последовательности:
*   **[Sequence Diagram: Daily Report Flow](../docs/diagrams/task-tracker-system/task-tracker-scheduler/sequences/daily-report-flow.md)**

### Ключевые Принципы Надежности
*   **Уникальный запуск:** Гарантируется `ShedLock`.
*   **Возобновляемость (Продюсер):** Прогресс (курсор пагинации) сохраняется в Redis.
*   **Гарантии доставки:** Система реализует семантику "как минимум один раз" (`at-least-once`).
    *   **Продюсер:** При сбое отправки в Kafka джоба аварийно завершается без сохранения прогресса, что приводит к повторной отправке при следующем запуске.
    *   **Консьюмер:** При сбое обработки используется механизм ретраев с последующей отправкой в Dead Letter Topic (DLT), чтобы не блокировать обработку.

## Взаимодействие с Другими Сервисами
*   **`task-tracker-backend`:** Обращение к внутренним эндпоинтам (`/api/v1/internal/**`) с аутентификацией по статическому API-ключу
(согласно **[ADR-0039](../docs/adr/backend-service/2025-06-18-21-50-безопасность-межсервисного-взаимодействия.md)**,
 **[ADR-0041](../docs/adr/backend-service/2025-06-18-22-00-api-для-взаимодействия-с-планировщиком.md)**).
*   **`Redis`:** Используется для `ShedLock` и хранения состояния джоб.
*   **`Kafka`:** Используется как транспорт для событий и команд между Продюсером, Консьюмером и `email-sender`.

## Конфигурация

Ключевые параметры конфигурации управляются через `application.yml`. Пример:

```yaml
app:
  scheduler:
    # --- Backend HTTP Client ---
    backend-client:
      url: "http://task-tracker-backend:8080"
      api-key: "${BACKEND_API_KEY}" # Обязательная переменная

    # --- Настройки джобы "Ежедневный отчет" ---
    jobs:
      daily-task-reports:
        enabled: true
        job-name: "daily-task-reports-v1"
        cron: "0 */15 * * * *" # Запуск каждые 15 минут для отказоустойчивости
        page-size: 1000
        kafka-topic-name: "task_tracker.scheduler.events.user_selected_for_report"

    # --- Настройки консьюмера "Ежедневный отчет" ---
    consumers:
      daily-report:
        enabled: true
        topic-name: "task_tracker.scheduler.events.user_selected_for_report"
        sink-topic-name: "task_tracker.reporting.events.daily_report_generated"
        group-id: "scheduler-daily-report-group"
        concurrency: 3
        max-poll-records: 500
```

*Полный список параметров см. в `SchedulerAppProperties.java`.*

## Observability
Сервис полностью интегрирован в стек обсервабилити:
*   **Логи:** Пишутся в структурированном JSON (ECS) формате.
*   **Трейсинг:** Используется OpenTelemetry для сквозной трассировки, включая асинхронные операции (Kafka, `@Scheduled`).
*   **Метрики:** Предоставляются кастомные метрики для мониторинга здоровья и производительности джоб и консьюмеров (см. `Metric.java`).

## Локальный Запуск
1.  Убедитесь, что все зависимости (PostgreSQL, Kafka, Redis, `task-tracker-backend`) запущены через `docker-compose up -d` в корне проекта.
2.  Установите переменную окружения `BACKEND_API_KEY` (значение должно совпадать с ключом, настроенным в `task-tracker-backend`).
3.  Запустите главный класс `TaskTrackerSchedulerApplication.java`.
